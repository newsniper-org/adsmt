# You were right twice — the wall was never the trace (rc.34.4)

Your isolation was exact. rc.34.3 fixed the wire (3.5 MB → 99 B) and the
comparison, but `jit_trace_digest` still rebuilt the **whole**
prelude∪query canonical clause set — CNF-flatten + sort + de-dup the
DIMACS of thousands of prelude clauses — on **every** `(check-sat)`. The
digest dropped the polynomial layer; it never touched the flatten+sort+
de-dup underneath, which is exactly the `(3) − (2) ≈ 0.38 s` you
measured. The prelude is fixed across a session, so that's redoing the
prelude's share every query. Fixed in **rc.34.4** with the lever you
proposed: precompute the prelude's order-independent fold once, combine
with the per-query delta each `(check-sat)`.

## What landed

**1. The digest is now an order-independent clause-fold.** Instead of
serialising a single global canonical clause set and hashing it, each
clause is hashed independently and the hashes are combined into a
multiset accumulator:

- `clause_name_hash(clause)` — a canonical K12-256 of one clause, keyed
  by **atom NAME** (sorted+de-duped `(name, polarity)` pairs,
  length-prefixed), **not** a global DIMACS index. Name-keying is the
  whole trick: a clause's hash no longer depends on the rest of the
  formula's atom set, so a prelude clause hashes the same whether or not
  the query added atoms that would have shifted the global index map.
  That's the property rc.34.3's global-index DIMACS lacked, and the
  reason the prelude couldn't be precomputed there.
- `clause_set_fold(clauses) -> (sum: [u8;32], count: u64)` — the per-
  clause hashes added **mod 2²⁵⁶** (AdHash), plus the clause count.
- `combine_fold` is exact: `combine(fold(P), fold(Q)) == fold(P ⊎ Q)`.
- `fold_to_digest((sum, count)) = K12(sum ‖ count_le)` → the 32-byte
  digest.

**On the combiner: I used modular addition, not XOR.** Your message said
"XOR / sum"; I went with sum deliberately. XOR self-cancels a duplicated
clause (`h ⊕ h = 0`, so `{C, C, D}` would alias onto `{D}`) and is linear
over GF(2) (sub-multiset collisions fall out of Gaussian elimination).
The digest is **soundness-critical** — an exact match makes the consult
trust a recorded verdict — so I wanted a multiset hash that's
collision-resistant under the random-oracle assumption (AdHash), not one
with a linear structure. Same order-independence, no self-cancellation.

**2. The prelude fold is precomputed into the bank at `--aot-bake`.**
New trailing **v1.3** field on the CDCL section:

```
CdclSection::prelude_clause_fold: Option<([u8; 32], u64)>
```

`at_end()`-gated exactly like rc.28's `had_opaque` (written only when
`Some`, so a bank without it is byte-for-byte the v1.2 layout, and v1.0–
v1.2 readers stop before it). `build_cdcl_section` computes it via the
now-`pub` `adsmt_engine::solver::clause_set_fold(clauses.iter())` — the
same code path the consult folds with, so they're byte-identical. On
load, `restore_cdcl_state_into` sets `Solver::aot_prelude_clause_fold`
from the bank value, or **recomputes it once** from the reconstructed
prelude for banks that predate the field (so older banks keep working,
just paying the one-time `O(prelude)` fold at load instead of an `O(1)`
read).

**3. The consult is now `O(query delta)`.** Per `(check-sat)`:

```
digest = fold_to_digest(combine(prelude_fold, fold(query-delta clauses)))
```

The prelude fold is the precomputed value; only the per-query
(non-prelude) assertion clauses are folded live. The cached prelude is
counted **exactly once** — in your flow the prelude lands in both the
assertion ledger and the clause cache, so the per-query pass skips any
assertion `Term` already in the cache (`aot_prelude_term_set`).

**`.lutrace` is unchanged** — still v2. The 32-byte digest is computed
differently but stored identically, so your bake script and the slim/
full emit paths don't change at all. The bank format gained the v1.3
field; re-bake picks it up automatically (your `aot-bake-prelude.sh`
needs no change — the field is written whenever `--aot-include-cdcl` is
set, which the §3.5 bake already passes).

## Soundness

The fold is an exact multiset homomorphism, so the incremental digest is
**byte-identical** to a from-scratch fold of the whole clause multiset —
there's no approximation. An exact digest match still means the same
clause multiset (AdHash collision-resistant ⟹ distinct multisets hash
distinct w.h.p.), and the consult still trusts a replayed Unsat only on
that exact match. The precompute is a pure performance restructuring of
a quantity that's verified equal both ways by the regressions below.

## Tests

5 new engine regressions + 1 bank round-trip:

- `clause_fold_is_an_exact_multiset_homomorphism` — `combine(fold(P),
  fold(Q)) == fold(P ⊎ Q)` exactly, and abelian.
- `jit_trace_digest_incremental_equals_whole_formula` — the live
  incremental digest equals a from-scratch whole-formula fold.
- `jit_trace_digest_counts_cached_prelude_once_not_twice` — the §3.5
  both-sources case folds the prelude exactly once.
- `precomputed_prelude_fold_matches_recompute` — using the stored fold
  == recomputing it.
- `restore_cdcl_state_into_picks_up_or_recomputes_prelude_fold` — load
  prefers the bank value, falls back to recompute.
- `cdcl_section_without_v1_3_fold_reads_back_none` — a v1.2-layout bank
  round-trips with `prelude_clause_fold == None`.

CLI-verified end-to-end: bake (`--aot-include-cdcl` → 317 B bank carrying
the v1.3 fold) → `--aot-load` reads it without error → slim emit (99 B
digest via the incremental path) → `--aot-load` + `--jit-trace-load` →
`unsat` via the exact-match short-circuit. Workspace **1074 → 1080**
green, 0 warnings.

## Honest scoping (yours, still valid)

Your note stands: each *distinct* obligation is a different formula →
digest miss → the consult canonicalizes the query delta, misses, and
falls through. So O(delta) mainly helps the **exact re-run** case —
re-verifying unchanged code against a warm bank — which is exactly what
§3.5.J targets. What rc.34.4 buys there: the consult's compute term is no
longer `O(#prelude)`; on a miss you now pay only the query-delta fold,
not a full prelude re-canonicalize. The break-even for the short-circuit
win drops from "search > ~0.38 s" to "search > query-delta fold," which
is essentially free for a typical small query.
