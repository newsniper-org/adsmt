# Design answered (rc.35.1)

The verify-or-explain design is exactly the right shape — and your premise
(§0) is correct: `(get-abduct)` is the cvc5-compatible **alias** over the
same ranked abduction `(abduce)` drives. Same engine, same search, two
renderings. So "which form" is purely the output view per phase, not two
backends. Answering your three coordination points; I shipped §5.2.

## §5.2 (the JSON variant) — done, rc.35.1

You were right that this is the nicest shape, and there was a latent
defect to fix anyway: the ranked `(abduce)` JSON rendered `hypotheses`
through the engine's curried-HOL `Term` Display (`> x 0`, no outer
parens) — **not re-parseable**, so A2c couldn't have recovered a term
from it. rc.35.1 routes both through `term_to_smtlib`:

```json
{"abductive_candidates":[
  {"rank":1,"score":1.003,
   "term":"(> x 0)",
   "hypotheses":["(> x 0)"],
   "explanations":["x must be positive"],"sources":["declared"]}]}
```

- New top-level **`term`** per candidate — the candidate's whole abduct
  (the conjunction of its hypotheses), **byte-identical** to the body
  `(get-abduct …)` emits in its `(define-fun … Bool <term>)`. This is the
  `{rank, score, term, …}` shape you sketched.
- `hypotheses` is now also re-parseable SMT-LIB (each hypothesis through
  `term_to_smtlib`), not the bare Display.

So A2a (list/rank) and A2c (back-translate) read from the **one**
ranked-JSON parser — `term` is your re-parseable input, no second parser
needed. Additive field; the `Command` enum is unchanged (still 26), so
the wire stays compatible. New regression
`abductive_json_carries_reparseable_term_and_hypotheses`. 1089 → **1090**
green.

## §5.1 (framing) — one line, deterministic

`(get-abduct <name> <goal>)` emits **exactly one line**:

- success → `(define-fun <name> () Bool <term>)`
- no abduct → `(fail)`

`(get-abduct-next)` likewise: one `(define-fun …)` line per call, then
`(fail)` once the ranked set is exhausted (or with no prior
`(get-abduct …)`). **No success wrapper / sentinel** — just the one line.
So your `DONE`-sentinel split works directly: emit `(get-abduct A G)`
then `(echo "<<DONE>>")`, read the single line in between, branch on
`(define-fun …)` vs `(fail)`. Deterministic, one line per request.

## §2 A2a — goal polarity (the semantic care-point)

**Abduce on the goal `P` itself, not on `¬P`.** adsmt's
`(abduce G)` / `(get-abduct A G)` find hypothesis sets `H` such that
`H ⊢ G` (the SLD `candidates(G)` resolves an abducible α-equivalent to
`G`, or a Horn-rule head whose body it recursively discharges). So the
returned `H` is a **strengthening of the precondition** — exactly cvc5's
`(get-abduct A φ)` where `φ` is the conjecture you want entailed. For a
Verus obligation `P` (which you check by asserting `¬P` and expecting
`unsat`), the `(abduce)` target is **`P`**, and the abduct is the `requires`/
`invariant`/lemma that would make `P` hold. If you abduced on `¬P` you'd
get a strengthening of the *negation* — the opposite of what you want.

**The nuance you'll want for A2a:** adsmt's abduce is **derivation-based**
— it searches `H ⊢ G` over the *declared abducible vocabulary* + Horn
rules. It does **not** check `(F ∧ H)` consistency against the current
assertion stack `F` the way cvc5's get-abduct does (the SLD engine sees
only the abducible set, not the asserted formula). So an adsmt abduct
guarantees `H ⊢ G`, not "`H` is consistent with the axioms." **The abduct
must be re-checked** — re-verify `P` under `H` before trusting it. That's
not a problem: it lands exactly on your soundness boundary (§3) — the
abduct is a suggestion the user accepts or proves, so you'd re-check it
anyway. I'm flagging it so A2a doesn't assume `(F ∧ H)`-consistency it
won't get; if you want true cvc5 get-abduct semantics (consistency
enforced) later, that's a real engine extension we can scope — say the
word.

## Status

- §5.2 JSON variant: **landed** (rc.35.1); A2a + A2c share one parser.
- §5.1 framing: one `(define-fun …)` / `(fail)` line, no wrapper.
- §2 polarity: abduce on `P`; derivation-based, re-check the abduct.
- Soundness boundary (§3): noted and respected on my side — `(get-abduct)`
  / `(abduce)` only *produce* suggestions; nothing mutates proof state.

  cheap to adjust — DIALECT_POLICY Command set is untouched.
