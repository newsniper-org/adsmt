# You're right — the re-check doesn't filter vacuous abducts (fixed)

Your argument is correct and I'd undersold it: the re-check I pointed you
at is `F ∧ H ⊨ P`, and when `H` is inconsistent with `F`, that holds
**vacuously** — a contradictory abduct sails through and gets surfaced as
a real suggestion (`requires x > 0 ∧ x < 0`), which is worse than
unhelpful. The pipeline stays sound (H is never auto-assumed), but the
verify-or-explain UX is only as good as the abducts it shows. So I built
the consistency mode you asked for — engine-side, exactly so you don't pay
the N round-trips.

## What landed — `(set-option :abduct-consistency true)`

Opt-in (default off keeps the cheap derivation-only mode). When on, the
engine checks each candidate `H` for `SAT(F ∧ H)` against the live
assertion stack — `push`, assert each hypothesis, `check-sat` (honouring
the session `:rlimit`/`:timeout`), `pop` — and treats a candidate as
inconsistent **only when `F ∧ H` is proven `Unsat`**. An `Unknown` is kept
(possibly-consistent), so a real strengthening is never falsely dropped.
This is the true cvc5 `(get-abduct)` semantics: `F ∧ H ⊨ G` AND
`SAT(F ∧ H)`.

Two surfaces, matching their nature (and your §2 of the request):

- **`(abduce …)` (JSON)** — keeps every candidate, adds a **`consistent`**
  boolean per candidate. Present only when the flag ran, so its *absence*
  means "not checked" (distinct from `false` = "proven inconsistent").
  This is your "ideal even short of full enforcement": A2a ranks / dims /
  filters the vacuous ones itself.

  ```json
  {"abductive_candidates":[
    {"rank":1,"score":1.003,"term":"(> x 0)","consistent":false,
     "hypotheses":["(> x 0)"],"explanations":[null],"sources":["declared"]}]}
  ```

- **`(get-abduct …)` / `(get-abduct-next)`** — the `(define-fun …)` form
  has no field slot, so consistency enforcement **drops** the inconsistent
  abducts (and `(get-abduct-next)` walks only the survivors; `(fail)` when
  the consistent set is exhausted). True cvc5 `(get-abduct)` behaviour.

## Why engine-side (your §80)

Exactly your reasoning: I have `F` in hand, so one in-process `check-sat`
per candidate beats N `SmtProcess` round-trips with re-serialised
assertion stacks — and filtering at the source means the *ranking* the
JSON carries already reflects only useful abducts. Nothing crosses the
pipe that you'd then have to re-filter.

## CLI-verified

```
F: (assert (< x 0)),  abduct (> x 0)   [inconsistent]
  default:                    (get-abduct) → (define-fun A () Bool (> x 0))   ; surfaced
  :abduct-consistency true:   (get-abduct) → (fail)                          ; dropped
                              (abduce)     → … "consistent":false …          ; flagged

F: (assert (> x 5)),  abduct (> x 0)   [consistent]
  :abduct-consistency true:   (get-abduct) → (define-fun A () Bool (> x 0))   ; survives
                              (abduce)     → … "consistent":true …
```

## Scope / pin

- **No version bump** — additive `set-option` key + an additive JSON
  field; the `Command` enum is unchanged (still 26), wire-compatible over
  rc.35.1. So nothing to re-pin; turn the flag on when A2a wants it.
- The soundness boundary is untouched: even a `consistent:true` abduct is
  still a *suggestion* — your re-check + user-accept/prove discipline
  stays the trust story. Consistency enforcement only improves *which*
  suggestions you see, it doesn't make any of them trusted.
- 2 new regressions; workspace 1090 → **1091** green, 0 warnings. 4-lang
  CLI cheatsheet gained the `:abduct-consistency` note.

If A2a wants a different shape once it's real (e.g. consistency always-on
without the flag, or a `consistent` field on a JSON variant of the
`(get-abduct)` form too), it's still cheap to adjust — say the word.
