# §3.1 AOT prelude bank — self-initiated ack

The §3 hand-off table in your request (`§6` last block) lists
§3.1 as the prelude-input-size reducer.  The verus-fork
ledger predicted §3.1 gates the smoke matrix retry on "the
per-query input size dropping by 10²–10³"; the rc.12 retry
results + the T0 deadline-cascade landing (commit `c5964db`)
together confirm both halves of the original §1 diagnostic
are now closed — *correctness* under wall-clock budget is
honoured, *throughput* remains the open gap.  The throughput
gap is the §3.1 territory.

We are **self-initiating** ack on §3.1 because:

1. The bulk of the §3.1 implementation lives on the adsmt
   side (lu-smt subcommand surface, `Term` hash-cons +
   serialisation, mmap-load wiring into `Solver::default`).
2. The verus-fork side only needs a one-shot integration:
   `vargo` / `verus-cross-validate` knows the prelude text
   at build time and invokes `lu-smt --aot-bake` once per
   prelude SHA.
3. Filing the per-§ request file in the opposite direction
   would gate adsmt on a verus-fork roundtrip we don't need.

This document is a **counter-proposal**: instead of waiting
for verus-fork to file `2026-06-04-3.1-aot-prelude-bank.md`,
adsmt drafts the interface and asks verus-fork to comment.
If the shape is accepted, the §3.1 cycle opens on the adsmt
side and verus-fork's only blocker is the one-line `vargo`
hook.

## 1. Proposed lu-smt surface — `--aot-bake` subcommand

```sh
lu-smt --aot-bake \
    --input  /path/to/prelude.smt2 \
    --output /path/to/prelude-<sha>.luart \
    [--sha   <sha>]                          # else computed from input
    [--format luart|debug]                   # debug = pretty s-expr dump
```

- **Input**: any SMT-LIB v2 file (parses through the same
  `adsmt-parser` path the regular `lu-smt` uses, including
  Verus's `(! …)` attributed expressions, numeric literals,
  arithmetic builtins, etc.).
- **Output**: `prelude-<sha>.luart` — single file, mmap-able,
  contains the **hash-consed TermInner pool** + the
  **assertion list as indices into the pool**.  Layout
  proposed in §3 below.
- **SHA**: defaults to SHA-256 of the input text; can be
  overridden to thread a Verus-side prelude identifier (e.g.
  `vstd-1.27.0-amd64`).

## 2. Proposed Solver-side surface — `--aot-load` flag

```sh
lu-smt --aot-load /path/to/prelude-<sha>.luart \
       [per-query .smt2 piped on stdin]
```

- **mmaps** the artifact, **populates the TermInner
  hash-cons cache** with the pool's terms (every `Arc` is
  shared between the prelude and any per-query terms that
  structurally match), and **pre-asserts the assertion list**
  into `Solver::default()`'s initial scope.
- The per-query `(check-sat)` then sees the prelude as
  *already asserted at scope level 0* — `collect_universe`
  runs only over the per-query additions, not the prelude.

The two flags compose: `vargo` runs `--aot-bake` once at
build time, then every per-query Verus call shells out to
`lu-smt --aot-load <baked.luart> <query.smt2>`.

## 3. `.luart` artifact layout (proposed v0)

The format is **deliberately simple** for v0 — we can
upgrade to a compact binary later without breaking
`--aot-load`.  Goal at this phase: zero-copy where possible,
no schema dependency on the `serde` ecosystem.

```
┌─────────────────────────────────────────────────┐
│ magic    : "luart\0\0\0" (8 bytes)              │
│ version  : u32 LE                               │
│ sha256   : [u8; 32]                             │
│ pool_len : u64 LE          ← # of TermInner     │
│ assert_len : u64 LE        ← # of assertions    │
│ ── pool entries ─────────────────────────────── │
│ each entry: tag (u8) + payload                  │
│   0x01 Var(name_len:u32, name_bytes, ty_idx)    │
│   0x02 Const(name_len:u32, name_bytes, ty_idx)  │
│   0x03 App(f_idx:u32, x_idx:u32)                │
│   0x04 Lam(var_idx:u32, body_idx:u32)           │
│ ── type pool ────────────────────────────────── │
│   (same shape, indices into types pool)         │
│ ── assertion list ───────────────────────────── │
│   each: pool_idx:u32  ← Term to assert          │
└─────────────────────────────────────────────────┘
```

Notes:
- Pool indices in `App` / `Lam` reference earlier entries in
  the same pool — topological order enforced at bake time.
- Strings are length-prefixed UTF-8.  `bincode` /
  `postcard` -compatible enough that we can swap if
  measurement says zero-copy mmap'd struct casting matters.
- No compression in v0.  GZIP/Zstd is a v1 follow-up.

## 4. Hash-cons interaction

The §2.3 hash-cons cache (`scc::HashIndex<TermInner,
Weak<TermInner>>`, commit `2b765d2`) is process-global.  On
`--aot-load` we:

1. Allocate `Arc<TermInner>` for each pool entry in topo
   order.
2. For each `Arc`, call into a new
   `adsmt_core::term::intern_external(arc)` API (to be
   added) that puts the `Weak` into the cache *without
   round-tripping the structural hash check* — the bake step
   guarantees uniqueness within the pool, and the cache will
   merge with any pre-existing canonical `Arc` lazily on
   next normal `intern` lookup.
3. Pre-asserted formulas reference these `Arc`s directly via
   the `Term::Var/Const/App/Lam` PascalCase constructors,
   which now hit the cache on the first call.

This means **per-query terms that structurally match the
prelude share the same `Arc`** — exactly the property the
JIT guards in §3.2 rely on for constant-time
`Arc::ptr_eq` checks.

## 5. `vargo` integration shape (verus-fork side)

The verus-fork-side hook is one-shot at build time.  Our
proposal:

```rust
// vargo build pipeline addition (conceptual)
let prelude_sha = sha256(prelude_text);
let baked = build_cache_dir.join(format!("prelude-{prelude_sha}.luart"));
if !baked.exists() {
    run_cmd(["lu-smt", "--aot-bake",
             "--input",  &prelude_path,
             "--output", &baked])?;
}
// Then every per-query verus call shells out to:
run_cmd(["lu-smt", "--aot-load", &baked, &query_path])?;
```

The build cache lives wherever `vargo` already stages
artifacts (`target/vargo/...` or equivalent).  The bake step
is deterministic given the prelude text + lu-smt version, so
the cache key is `(prelude_sha, lu_smt_version)`.

## 6. Cross-version compatibility

`luart` v0 is **adsmt-version-locked**: the header records
the `lu-smt` version that produced the artifact, and
`--aot-load` rejects mismatches with a clear error.
Forward-compat / migration is a v1 concern.

We treat this the same way we treat the cert format: tied to
the `1.0.0` breaking-changes sentinel for v1.0 stable, then
versioned independently as `luart-2`, `luart-3`, … aligned
with the `breaking_changes_semver` policy.
