# You caught my regression — fixed (rc.34.6)

You're right, and thank you for catching it. rc.34.5 hit the §3.5.J
consult goal (`(3) − (2) ≈ 0`) but I built the prelude atom map in
`with_aot_cdcl`, which runs on **every** `--aot-load` — so the AOT-only
path paid a prelude-scale build for a map only `replay_events` (the trace
path) ever reads. That quietly took back the one §3.5 win that was already
net-positive. My mistake: I gated the precompute on the *bank* loading,
not on the *consumer* (a trace) being present.

## The fix — gate on a loaded trace

I took your gate (the cleaner of the two): the precompute moves out of
`with_aot_cdcl` and into **`set_loaded_jit_trace`** — it's built only when
a trace is actually installed.

```rust
pub fn set_loaded_jit_trace(&mut self, trace: adsmt_jit::CdclTrace) {
    self.loaded_jit_trace = Some(trace);
    let base = self.build_prelude_atom_map();   // only reached with a trace
    self.aot_prelude_atom_map = Some(base);
}
```

A bare `--aot-load` never calls `set_loaded_jit_trace`, so
`aot_prelude_atom_map` stays `None` and nothing prelude-scale is built —
`verify-adsmt-fast` is back to the plain bank-reconstruction cost.

New `Solver::build_prelude_atom_map` builds from the **fixed** prelude
sources only — `aot_pool_terms` + the prelude clause atoms + the flatten
of the prelude *assertion* terms (those in `aot_prelude_term_set`) — with
**no** per-query assertions. That's deliberately stricter than "read
`all_assertions()` and trust it's prelude-only at install time": it stays
the prelude base even in your future "trace loaded mid-session" case.

In the lu-smt driver `with_aot_cdcl` (bank) always runs before
`set_loaded_jit_trace` (trace), so the pool + clauses + prelude assertions
are all in place when the build runs. And if the base is ever absent at a
consult (unusual ordering), the consult falls back to the full
`live_atom_map` build — correct, just without the speedup. So the gate is
a pure perf change, never a correctness one.

## Where the cost lands now

Exactly the amortization you described: the `O(prelude)` build is paid
**once per JIT session** (at trace install), not per `(check-sat)` and not
on AOT-only. In your real streaming flow (one `--aot-load`, many queries
in a session) it's negligible per query.

One honest note on the micro-benchmark: in your *single-query* `(2)`/`(3)`
process measurement the one-time build now shows up in `(3)` (trace path)
and not `(2)` — so `(2)` drops back to the fast AOT-only number, but
`(3) − (2)` for a single query will show the build again. That's the
measurement artifact of single-query processes, not a per-consult cost —
in a multi-`(check-sat)` session the build amortizes to ~0 per query while
`(2)` stays fast. The thing that matters — AOT-only un-taxed, per-consult
work O(query delta) — holds.

My synthetic 4002-clause prelude after the fix:

```
(1) baseline:                      ~1.6  ms
(2) --aot-load ONLY (no trace):    ~32   ms   (bank reconstruction only —
                                               the ~0.40 s build is gone)
(3) --aot-load + --jit-trace-load: ~36   ms   verdict=unsat
```
