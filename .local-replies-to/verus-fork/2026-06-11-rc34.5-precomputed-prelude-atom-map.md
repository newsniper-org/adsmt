# You found the last one — `live_atom_map()`, exactly (rc.34.5)

Your diagnosis was precise and correct. I read the path you pointed at:
the digest fold *is* O(delta) now, `canonical_gf2_signature` *is* properly
guard-gated and skipped — and one line down, `live_atom_map()` rebuilt a
hash→`Term` map over the **whole** bank ∪ per-query formula on every
consult (re-flatten + `to_string` + hash thousands of prelude atoms). It
was the same whole-formula sweep the digest used to be, moved one concern
over — resolving `replay_events`' recorded content-hash atoms (the rc.34.1
`diverged` fix). That's the 0.38 s.

I took **lever 2** — precompute the prelude's share, the same treatment
the digest fold got — because it also covers full traces and the §3.5.C
cross-query case, not just slim.

## What landed

**1. The prelude atom map is precomputed once at `--aot-load`.** New
field:

```rust
Solver::aot_prelude_atom_map: Option<(HashMap<u32, Term>, bool)>
```

built at the tail of `with_aot_cdcl`. At that point `all_assertions()` is
the prelude only (no per-query assertions yet), so the existing full
`live_atom_map()` there yields **exactly** the prelude base — pool terms +
prelude clause atoms + prelude assertion atoms — plus its internal
collision flag. One O(prelude) build, at load, not per consult.

**2. The consult chains a small per-query map over the base.** New helper
`query_atom_map(base)` flattens **only** the non-prelude assertions
(prelude `Term`s, the ones in `aot_prelude_term_set`, are skipped —
re-flattening them is precisely the prelude cost this removes) and
cross-checks each atom's hash against `base`: an already-in-base hash
isn't duplicated; a same-hash / different-term landing flips the collision
flag. The replay then resolves through the chain:

```rust
crate::cdcl::replay_events(&trace.events, |i| {
    qmap.get(&i).or_else(|| base.get(&i)).cloned()
})
```

To thread that with **no clone of the prelude map**, `replay_events`
changed from `atom_map: &HashMap<u32, Term>` to
`resolve: impl Fn(u32) -> Option<Term>`. The `None` arm (no bank loaded,
or a test-built solver) falls back to the full whole-formula
`live_atom_map()` build, so nothing off the `--aot-load` path changes.

**3. Slim traces never touch the prelude — at all.** Exactly your point:
a slim/digest trace's events are `[Restart, Conflict@0]`, which reference
no atom, so `replay_events` never calls `resolve`. The per-consult work
is then purely the per-query delta (digest fold + query atom map), end to
end.

## Soundness / parity

- **Coverage is identical.** The chain resolves an atom iff it's in the
  query map or the base — the same set the old merged map carried — so
  `diverged` and `root_conflict` are unchanged. (New regression
  `query_atom_map_skips_prelude_and_chained_resolver_matches_full` checks
  the chain resolves every atom the full rebuild does, atom-for-atom.)
- **Collision parity.** Total collision = base-internal (computed once at
  load) ∪ query-internal ∪ query-vs-base (per consult) — the same set the
  old whole-formula `live_atom_map` reported. The one observable
  difference is which `Term` wins on a hash collision (query-first chain
  vs first-insert merge) — but a collision *disables* the term-dependent
  `level0_falsifies_prelude_clause` backstop, and the exact-match verdict
  is term-independent, so it's sound either way.
- **No wire/bank/`.lutrace` change.** This is purely the in-engine
  atom-map build. The v1.3 bank field and the v2 `.lutrace` are untouched;
  your bake script needs no change.

## Numbers

I don't have your verus prelude locally, so I built a synthetic
4002-clause / 2000-var prelude with an unsat core and reproduced your
`(3) − (2)`:

```
(2) --aot-load only:               ~40 ms
(3) --aot-load + --jit-trace-load: ~33 ms
consult marginal (3) − (2)       ≈ 0 ms   (within noise; the consult
                                            short-circuits faster than the
                                            native solve it replaces)
```

At rc.34.4 even this 4002-clause prelude would have re-flattened + re-
hashed all of it per consult inside `live_atom_map`. Now the consult
marginal doesn't scale with the prelude at all.
