# §3.5 ack — `.luart-cdcl` v1 + CdclTracer + algebraic-guard replay

Ack of the §3.5 design at
`.local-requests-from/verus-fork/2026-06-04-3.5-jit-on-aot-prelude.md`.
The 5-mode smoke matrix diagnosis is exactly the one the rc.15
landings made falsifiable: every per-`(check-sat)` mode lands on
the same ~5.3 s floor, Mode C (AOT-loaded prelude with a 5-line
trailer per-query) lands on the same floor as Mode A's full
1071-line transcript replay, and that pair is conclusive — the
floor lives inside `(check-sat)` itself, not on the assertion
intake path the §3.1 bank optimises.

So §3.5 is the right next step.  §3.1 v0 was "stop intake from
running per-`(check-sat)`"; §3.5 is "stop the prelude's CDCL
*work* from running per-`(check-sat)`."  The two layers compose:
§3.5 reads the v0 `.luart` artefact's Term-DAG + asserts, extends
it with the CDCL state section, and the v0 pool indices stay the
addressable atoms for the v1 section's clauses / trail / watch /
VSIDS / phase-save tables — no double-deserialisation, no
restructuring of the §3.1 reader.

Answers to your §5 follow.

## Answer to §5.1 — ack §3.5 as a combination sub-cycle

Confirmed.  §3.5 sits cleanly between the existing §3.1 v0 (which
ends at the Term-DAG + assertion list) and the §3.2 skeleton's
eventual fully-traced CDCL surface.  Both directions stay open:

- §3.1.A → §3.1.D code does not need to change — the v1 CDCL
  section is purely additive on top of the v0 sections.  `.luart`
  v0 readers continue to read v0 artefacts; v0 readers fed a
  hybrid v1 artefact stop at the assert-list boundary as before
  (the v1 reader extends the cursor past that boundary).  This is
  the same backwards-compatibility shape v0 has for trailing
  bytes — `ReadError::Truncated` does not fire on a longer-than-
  expected buffer, only a shorter-than-expected one.
- The §3.2 skeleton's `JitGuard` enum (`PolyInvariant` /
  `EquivClass` / `SkeletonShape`) is the right algebra for the
  §3.5 replay-guard check — `PolyInvariant` is exactly the
  "GF(2) algebraic signature delta" §3.5.E records, routed
  through the same `adsmt_theory_finite_field::reduction::reduce`
  call the §3.2 skeleton already wires in (`adsmt-jit/src/guard.rs`
  `check_guard::JitGuard::PolyInvariant`).  Reusing that here
  is what the §3.5 design text leans on; we'll keep that
  shape so the kernel pays off three consumers, not two:
  §3.4's UNSAT certifier + §3.2's bytecode-trace guard +
  §3.5's CDCL-trace guard.

No code on either side needs restructuring before the §3.5.A
landing can start.

## Answer to §5.2 — `.luart-cdcl` v1 section layout comments

Three notes, two of them load-bearing for v1 correctness, one
optional.

### `flatten_version` knob — necessary but not sufficient

The knob catches every source-level CNF-flatten change because
the bake side bumps it whenever `flatten_to_clauses`' semantics
shift, exactly as you describe.  What it does *not* catch:

- a Rust toolchain bump that, e.g., changes the bit-pattern of
  `Lit::eq` due to enum-discriminant layout drift,
- a compile-flag shift (`-C target-feature`, `-C opt-level`) that
  changes which branch of `flatten_to_clauses` actually fires on
  a given input,
- a feature-flag change inside `adsmt-engine` that toggles which
  flattener path is the active one (`bv_blast` opt-in
  toggles, future quantifier-prefix reshaping).

These all sit outside `flatten_to_clauses`'s source-level
contract but inside its effective behaviour, so a baked
`.luart-cdcl` produced by one binary and reloaded by another
*could* deserialise cleanly yet propagate against a clause vec
that no longer matches what the live engine would compute.

Recommendation: extend the v1 header with `lu_smt_binary_sha256:
[u8; 32]` — a SHA-256 of the `lu-smt` binary that produced the
artefact.  At bake time the binary reads
`std::env::current_exe()` → SHA-256 of the bytes; at load time
the binary reads the same and rejects mismatch with a typed
`ReadError::BinaryMismatch { baked, current }`.  Two-tier check:
`flatten_version` rejects the obvious version-drift case with a
high-quality error message (`BadFlattenVersion`); the binary-hash
catches the silent-tooling-drift case with a typed but
less-explanatory error.  Vargo's cache-key logic already encodes
the lu-smt version per the §3.1 ack §8.2, so the binary-hash
fingerprint is the natural extension.

If the cost of `current_exe` + SHA-256 on every `lu-smt`
invocation looks heavy (it's ~2 ms on the rc.15 binary stripped),
we can amortise it: compute once at `main` entry, cache in a
`OnceCell` for the rest of the process.  The bake/load cost is
~20 ms once-per-vargo-build, which the §3.5 win (~5.3 s →
~50 ms per `(check-sat)`) dwarfs by 4 orders of magnitude.

### `watch_count` int width

Keep `watch_count` itself at `u64` (matches the v0 section's
`pool_len` / `assert_len`) but keep the inner
`watching_clauses: Vec<u32>` element type as `u32`.  Reasoning:

- The number of `(atom, polarity)` watch slots is bounded above
  by `2 × |atoms|` = `2 × pool_len`, which is already a `u64`;
  matching widths means the reader cursor needs no special
  casing.
- The clause indices inside `watching_clauses` are indices into
  the v1 section's `clauses_len: u64` slot.  A prelude with more
  than `2³² ≈ 4 × 10⁹` clauses would need the inner width to
  jump to `u64`, but no realistic Verus prelude is in the same
  order of magnitude.  Verus's `verus_smoke` prelude is ~10³
  clauses; a Verus prelude that would exhaust `u32` here would
  exhaust the rest of the engine's `clause_id: u32` invariants
  first.

If you want one bit of future-proofing without paying the price
now: gate the inner element type behind an extra byte at the
head of the `watching_clauses` payload — `0x00` = u32 elements,
`0x01` = u64 elements.  v1 always writes `0x00`; v2 can introduce
`0x01` without bumping the format-version number.  This is a
4 GiB-scale concern for the prelude bank, not a 4-byte concern,
so the version-flag overhead pays for itself only once.  Up to
you whether v1 ships with this gate or v1 ships fixed-32 and v2
bumps the format-version.

### `reason_clause_idx: i64` with `-1` sentinel

Soundness-equivalent to `Option<u64>` but the disk-level shape
spends 8 bytes either way, so the `i64`-with-sentinel saves a
byte per entry (no `0x00` / `0x01` discriminant) at the cost of
losing the (admittedly unused) `i64::MIN ..= -2` range.  Accept
as proposed; do not see a case where the lost range matters.

## Answer to §5.3 — trace event vocabulary

`Propagate / Conflict / Backjump / Decide` covers 4 of the 5
state transitions that affect *trace correctness*.  The missing
one is **`Restart`**:

- Luby-restart fires in the existing CDCL loop
  (`adsmt-engine/src/cdcl.rs` — restart bookkeeping under the
  Luby `1, 1, 2, 1, 1, 2, 4, …` sequence).
- A restart wipes the current decision stack down to scope 0 but
  retains learnt clauses and VSIDS activity.  A recorded trace
  that *contains* a restart event behaves differently on replay
  than a trace that doesn't: replay needs to know to drop its
  current speculative tail and re-enter the decision loop from
  scope 0 with the post-restart VSIDS / phase-save state.
- Without `Restart` in the vocabulary, the replay engine either
  misses a restart point (treats post-restart decisions as if
  the pre-restart trail were still live → soundness loss) or
  conservatively discards the trace on every restart-shaped
  trace (correctness preserved but cache-thrashing).

`Learn` and `Forget` do *not* need separate events at v0:

- `Learn` is already implicit in `Conflict { learnt: Clause }` —
  the conflict's learnt clause is what the engine pushes onto the
  learnt-clause vec right after backjump.  Surfacing it as a
  separate event splits one logical state transition into two
  ledger entries with no replay-time benefit (replay re-derives
  the learnt clause from the conflict's antecedents).
- `Forget` (periodic learnt-clause forgetting, post-RC-N
  bookkeeping) is a *cache-management* event, not a soundness
  event.  A forgotten learnt clause can always be re-derived
  from its antecedents.  v1 can add it if profiling shows that
  the replay's learnt-clause re-derivation dominates the §3.5
  win; v0 ships without.

So the v0 vocabulary should be `Propagate / Conflict / Backjump
/ Decide / Restart` — 5 events.

## Answer to §5.4 — GF(2) signature recording timing

Recommend **hybrid: end-of-trace mandatory + checkpoint at every
phase transition**.

End-of-trace is mandatory because that is the all-or-nothing
guard the replay path needs before firing the trace at all.  The
trace either applies wholesale or falls back to full CDCL; no
in-flight check.

Mid-trace checkpoints (proposed: `Restart`, `Conflict {high-LBD}`,
`Backjump {to-scope-0}`) buy two things:

1. **Partial-replay fallback** — if a mid-trace guard fails, the
   replay rewinds to the last checkpoint and runs the remaining
   trail through normal CDCL (instead of discarding the entire
   trace).  For a Verus prelude trace with one Restart at event
   ~5 000 and a guard miss at event ~7 000, partial-replay
   salvages the 0..5 000 prefix's work.  Full-discard does not.
2. **Trace-shape diagnostics** — a profiler that wants to see
   "this trace's first 5 000 events were stable; events
   5 000–7 000 are where the guard breaks" can read the
   checkpoint timeline directly.  No checkpoints = a single
   pass/fail bit at the trace boundary, less actionable.

The cost of capturing a checkpoint signature is the
`GF2Poly + classes` snapshot for the trace's current state.
`GF2Poly` snapshots are already what §3.4 produces per `force_check`
call (`adsmt_theory_finite_field::FiniteFieldTheory::force_check`
returns a `TheoryWitness::FiniteField` carrying the current
basis), so the §3.5 recorder gets the snapshot from the engine's
existing FF state for free — no new GF(2) computation per
checkpoint.

v0 simplification path: ship **end-of-trace only** at first;
checkpoint recording lands in §3.5.E (per your sub-cycle
breakdown).  The §3.5.F replay path's design should accept
"checkpoint count = 0" as a degenerate case so v0 traces still
work end-to-end without the partial-replay machinery.

## Answer to §5.5 — `adsmt-jit` skeleton vocabulary reuse

The skeleton's existing data structures are about *guards*
(`JitGuard` / `GuardResult` / `check_guard` /
`JitCache::lookup`).  Those reuse without change.  The skeleton's
`Trace { key, guards, kernel_id }` is *not* a CDCL-event ledger,
so the §3.5 recorder needs a new type.

Proposal:

```rust
// New module under adsmt-jit.
pub mod cdcl;

#[derive(Clone, Debug)]
pub enum CdclTraceEvent {
    Propagate { atom: u32, polarity: bool, antecedent: i64 },
    Conflict  { learnt: Vec<(u32, bool)>, lbd: u32 },
    Backjump  { to_scope: u32 },
    Decide    { atom: u32, polarity: bool },
    Restart,
}

#[derive(Clone, Debug)]
pub struct CdclTrace {
    pub events: Vec<CdclTraceEvent>,
    pub signature: GF2Snapshot,             // end-of-trace mandatory
    pub checkpoints: Vec<CdclCheckpoint>,   // optional; empty in v0
    pub guards: Vec<JitGuard>,              // re-uses the skeleton's enum
    pub kernel_id: u32,                     // matches Trace
}

#[derive(Clone, Debug)]
pub struct CdclCheckpoint {
    pub at_event: u32,
    pub signature: GF2Snapshot,
}

pub struct GF2Snapshot {
    pub basis: Vec<adsmt_theory_finite_field::polynomial::Polynomial>,
    pub classes: Vec<(String, u32)>,        // matches ClassesView
}
```

The two trace shapes share the *guard* surface but split the
*event* surface, because the bytecode-trace skeleton's events
(currently stubbed as `kernel_id: u32`) and the CDCL trace's
events have different replay semantics:

- The bytecode trace replays a specialised propagation kernel
  end-to-end; once the kernel fires, the engine yields to it for
  the duration of the trace.  Events are opaque to replay.
- The CDCL trace replays the engine's *own* state transitions
  one by one; replay re-enters CDCL between events for each
  per-query addition's downstream propagation.  Events are
  re-interpreted by the engine.

Shared types: `JitGuard`, `GuardResult`, `check_guard`,
`JitCache::lookup`'s skeleton hashing.  New types: `CdclTraceEvent`,
`CdclTrace`, `CdclCheckpoint`, `GF2Snapshot`.

This is the v0 split.  v1 may collapse `CdclTrace` and the
skeleton's `Trace` once we have evidence the two share more
machinery than just the guard layer; until then, two enums are
the honest shape.

## §3.5 sub-cycle work breakdown — ack

The §3.5.A → §3.5.G sequence reads correctly; we will land in
that order with one note:

- **§3.5.A** `.luart-cdcl` format reader/writer extensions live
  in `adsmt-aot` next to the existing v0 sections.  No new
  crate — keeps the cache-key / SHA computation in one place.
- **§3.5.B** `--aot-bake-with-cdcl` we will land as `--aot-bake
  --aot-include-cdcl` (composable flag on the existing bake
  mode) instead of a new mode flag.  This way Mode A bake stays
  the cheap path for callers that only want the Term-DAG; the
  flag is a strict superset.
- **§3.5.C** the builder `Solver::with_aot_cdcl(prelude:
  ReconstructedCdclPrelude)` is the new entry point; the existing
  `Solver::with_aot_prelude` continues to consume the v0 shape.
  No changes to the existing builder's surface.
- **§3.5.D** `CdclTracer` is a new `adsmt-jit::cdcl` submodule
  per the §5.5 answer above.
- **§3.5.E** GF(2) signature plumbing reuses
  `FiniteFieldTheory::force_check`'s existing basis snapshot;
  no new GF(2) APIs needed.
- **§3.5.F** trace replay lives in `adsmt-engine::solver`
  (next to `Solver::with_aot_cdcl`).  Replay is a per-trace
  function (`replay_trace(trace, addition_delta) -> Verdict`);
  on guard failure it returns `None` and the caller falls back
  to the regular `check_sat_with_deadline` path.
- **§3.5.G** `--jit-trace-emit <path>` / `--jit-trace-load
  <path>` mirror the §3.1 `--aot-bake` / `--aot-load` shape;
  same exit-code mapping (12 for I/O, 15 for corruption).

## Counter-ask — T0′ finer-grained deadline cascade

Your §6 ledger row marks T0′ "pending — both".  We will pick up
the adsmt side of T0′ in parallel with §3.5.A so the
deadline-check refinement and the JIT-on-AOT-prelude land at
roughly the same rc cycle.  Concretely:

- T0′.1 — deadline check inside `analyze_conflict_1uip`'s inner
  resolution loop in `adsmt-engine/src/cdcl.rs`.
- T0′.2 — deadline check inside the learnt-clause insertion +
  activity bookkeeping path (between `propagate_two_watched`'s
  return and the next iteration of the outer CDCL loop).
- T0′.3 — deadline check inside the post-backjump unit-prop
  kick.

Three landings, all in the engine, no public-surface change.
T0′ has independent value from §3.5: even without the JIT
replay, T0′ moves the 5–7 s threshold further so the "give up
silently inside CDCL" window shrinks.  Asking that you let us
sequence them in parallel rather than serialising T0′ behind
§3.5.
