# §3.5 — JIT-on-AOT-prelude

## 1. Why §3.5

The rc.15 5-mode smoke matrix retry (recorded in the §6 ledger of
the parent engine-refactor request) ships every layer §3.1 and
§3.2 currently scope:

- §3.1.A → §3.1.D — `adsmt-aot` crate + `lu-smt --aot-bake` /
  `--aot-output` / `--aot-sha` / `--aot-load` + `Solver::with_aot_prelude`
  + `intern_external`.
- T1.1 / T1.2 — `--finite-field-periodic` / `--finite-field-budget-exhaustion`
  CLI + mid-session `(set-option :finite-field-…)` SMT-LIB handler.
- §3.2 — `adsmt-jit` meta-tracing skeleton with algebraic guards.
- §3.3 — `adsmt-stalmarck` simple-rule pre-saturation skeleton.

…and yet **all five matrix modes hit the identical 5–7 s
threshold the rc.14 baseline already had** (Mode A = baseline;
Mode B = `--finite-field-budget-exhaustion`; Mode C =
`--aot-load`; Mode D = AOT + F4; Mode E =
`--finite-field-periodic 16`).  Mode C → Mode A invariance
(5 807 ms vs 5 221 ms at `--rlimit 1 s` — both `unknown` →
`(:reason-unknown "canceled")`) is the strongest signal we have
that the `~5.3 s` floor lives **inside `(check-sat)` itself**,
not in parse / declare / assert / CNF-flatten.

Two complementary observations narrow the gap:

1. **§3.1 v0 `.luart` stops at the Term-DAG**.  The artifact
   carries the hash-cons pool + assertion list, but neither the
   post-flatten clause vec nor the initial two-watched index nor
   the prelude-only BCP closure.  Every `(check-sat)` rebuilds
   those from scratch — even on a baked + loaded prelude.
2. **§3.2 traces nothing the smoke matrix hits**.  The skeleton
   records bytecode-level traces but has no hook into CDCL's
   propagation kernel, so the prelude's propagation closure is
   never observed as a trace candidate.

§3.5 is the *combination* sub-cycle that closes both gaps in one
shot: bake more of the post-`(check-sat)` state into the AOT
artifact, then JIT-trace the prelude's CDCL propagation closure
so per-query `(check-sat)`s replay the trace under a §3.4-style
GF(2) algebraic guard instead of re-running the prelude.

## 2. Architecture (three layers)

### 2.1 `.luart-cdcl` (proposed v1 `.luart` extension)

`.luart` v0 layout (per §3.1.A) ends at the assertion list.  v1
adds a **CDCL section** carrying the prelude-only state CDCL
would arrive at right after asserting every prelude clause and
running BCP to fixpoint without ever making a decision:

```
┌─ (v0 sections unchanged) ──────────────────────────────────┐
│ magic / version / sha256 / pool / type pool / asserts      │
└────────────────────────────────────────────────────────────┘
┌─ v1 CDCL section ──────────────────────────────────────────┐
│ flatten_version : u32 LE     ← which CNF flattener emitted │
│                                (so reload validates the    │
│                                 prelude clauses match the  │
│                                 current adsmt-engine)      │
│ clauses_len  : u64 LE        ← post-flatten clause count   │
│ ── clause entries ─────────────────────────────────────── │
│ each: lit_count:u32 + (atom_pool_idx:u32 polarity:u8)*    │
│ ── prelude BCP trail ──────────────────────────────────── │
│ trail_len    : u64 LE                                      │
│ each entry: atom_pool_idx:u32 + polarity:u8 +              │
│             reason_clause_idx:i64  (-1 = derived from      │
│             prelude only, no antecedent clause from per-   │
│             query side)                                    │
│ ── two-watched index ──────────────────────────────────── │
│ watch_count  : u64 LE                                      │
│ each entry: atom_pool_idx:u32 + polarity:u8 +              │
│             watching_clauses:Vec<u32>                      │
│ ── VSIDS initial activity ─────────────────────────────── │
│ vsids_count  : u64 LE                                      │
│ each entry: atom_pool_idx:u32 + activity:f64               │
│ ── phase-saving prelude polarities ────────────────────── │
│ saved_phase_count : u64 LE                                 │
│ each entry: atom_pool_idx:u32 + polarity:u8                │
└────────────────────────────────────────────────────────────┘
```

Notes:

- All atom references are indices into the `.luart` v0 pool, so
  loading the CDCL section piggybacks on the same `intern_external`
  the v0 Term-DAG already uses.  No double-deserialisation.
- `flatten_version` is bumped any time the engine's CNF
  flattener semantics shift in a way that would invalidate a
  baked clause vec — even an aliasing change in `flatten_to_clauses`
  rotates this number, so a stale `.luart-cdcl` fails fast at
  load time with a clear `BadFlattenVersion` error (mirroring
  `BadTypeString` / `BadMagic` from §3.1.C).
- `reason_clause_idx = -1` marks trail entries that were
  derived purely from prelude clauses with no per-query
  antecedent dependency — the JIT trace's guards check that
  these stay derived regardless of per-query additions.
- The two-watched section is the actual win for skipping
  prelude flatten: re-deriving it on every `(check-sat)` is
  ~`O(clauses × avg_clause_len)` work that the JIT trace can't
  shortcut on its own.

### 2.2 CDCL-loop trace recording (§3.2 refinement)

`adsmt-jit` 's meta-tracing skeleton grows a `CdclTracer` that
hooks the existing CDCL loop's main state transitions:

```
                                    ┌── trace event recorded ──┐
                                    │                          │
  ┌─ cdcl_solve_with_model ─┐       │                          │
  │ propagate_two_watched   ├──────►│ Propagate {atom, polar,  │
  │                         │       │            antecedent}*  │
  │ if conflict:            │       │                          │
  │   analyze_conflict_1uip ├──────►│ Conflict {learnt}        │
  │   backjump              ├──────►│ Backjump {level}         │
  │ else if no decision:    │       │                          │
  │   pick_vsids_atom       ├──────►│ Decide {atom, polar}     │
  │ else:                   │       │                          │
  │   restart / give up     │       │                          │
  └─────────────────────────┘       └──────────────────────────┘
```

Per the §3.2 design (GF(2) polynomial-relation + equivalence-class
semantic guards), the trace's recorded invariant is **not**
"these specific literals propagated to these specific values".
It is "the GF(2) relations and equivalence classes that held
when the trace was recorded continue to hold".  The trace stays
valid for any `(check-sat)` whose per-query additions don't
disturb those relations.

Concretely, a recorded trace carries:

- **Trace prefix**: the sequence of propagation / decision /
  conflict events the prelude state walked through in the
  recorded run.
- **Algebraic signature**: per-trace GF(2) polynomial relations
  and per-equivalence-class atom membership at trace-record
  time.  This is the same signature §3.4's `FiniteFieldWitness`
  carries.
- **Guard recipe**: for each trace event, a Boolean predicate
  over the algebraic signature that says "this event is still
  valid if these relations hold".

### 2.3 Trace replay at `(check-sat)`

At per-query `(check-sat)` time, with `--aot-load <luart-cdcl>
--jit-trace <trace-path>`:

```
1. Load `.luart-cdcl` v1 → restore Term-DAG, post-flatten
   clauses, BCP trail, watch index, VSIDS, phase save.
2. Assert per-query additions through the regular Solver
   path → the additions land at scope ≥ 1.
3. Compute per-query GF(2) algebraic signature delta against
   the baked signature.
4. If trace's guard recipe is satisfied:
     → replay trace events directly (zero CDCL search on prelude).
     → run only the per-query additions through the regular CDCL.
   Else:
     → discard trace.
     → fall back to full CDCL (current behaviour).
5. Emit verdict.
```

The trace is "valid" exactly when:

- Every algebraic relation in the trace's signature is preserved
  by the per-query additions, AND
- Every equivalence class is intact (no new merge or split).

These are checks at the §3.4 Gröbner-kernel level — constant-time
relative to the prelude size, polynomial in the per-query delta
size.  This is the layer that **only §3.4 can provide soundly**;
without it the trace replay would silently miss UNSAT verdicts
where a per-query addition refutes a prelude-derived literal.

## 3. Expected smoke matrix impact

Predicted post-§3.5 mode column (vs current rc.15 5-mode):

| mode | rlimit 1 s | rlimit 5 s | rlimit 7 s |
|---|---|---|---|
| **rc.15 baseline** | 5 221 ms / unknown | 5 352 ms / unknown | 60 002 ms / timeout |
| **§3.5 + AOT-CDCL** | ≤ 1 500 ms / *real verdict* | ≤ 1 500 ms / *real verdict* | ≤ 1 500 ms / *real verdict* |

`(check-sat)` cost breakdown for the verus_smoke fixture:

| step | current cost | §3.5 cost |
|---|---|---|
| parse prelude assertions | ~200 ms | 0 (skipped via `.luart` v0) |
| declare-sort / declare-fun | ~50 ms | 0 (skipped) |
| assert + intern Term-DAG | ~300 ms | 0 (skipped) |
| **CNF flatten on prelude** | **~1 500 ms** | **0 (skipped via `.luart-cdcl` v1)** |
| **initial BCP on prelude** | **~1 500 ms** | **0 (skipped via `.luart-cdcl` v1)** |
| **CDCL search on prelude** | **~1 800 ms (hits deadline floor)** | **~50 ms (JIT trace replay)** |
| CDCL search on per-query | ~negligible | ~negligible |
| **total** | **~5 350 ms** | **~50 ms baked-state + per-query** |

If the prediction holds, every `--rlimit ≥ 1 s` budget surfaces a
**productive** verdict (sat / unsat / abductive) on the
verus_smoke fixture for the first time — closing the smoke
matrix functionally, not just structurally.

## 5. What we ask of adsmt

In priority order:

1. **Acknowledge §3.5 as a combination sub-cycle** — confirm it
   sits between the existing §3.1 v0 (Term-DAG bake) and the
   §3.2 skeleton's eventual fully-traced CDCL surface, without
   restructuring either.
2. **Comment on the `.luart-cdcl` v1 section layout** — does the
   `flatten_version` knob suffice for staleness detection, or
   should we additionally hash the engine binary's CNF-flatten
   code path? Should `watch_count` use 32-bit pool indices like
   the rest of the format, or jump to 64-bit for the prelude
   sizes Verus will throw at it?
3. **Confirm the trace event shape** — `Propagate` /
   `Conflict` / `Backjump` / `Decide` covers what we'd want a
   trace to record; is there a state transition I'm missing
   (Restart? Learn? Forget?)?
4. **Comment on the GF(2) signature recording timing** — should
   the signature be captured at every trace event (more
   accurate, more cost) or only at the trace's end (cheaper, but
   replay guards have to validate the whole interval at once)?
5. **(Optional)** Confirm whether the §3.2 skeleton's existing
   trace-event vocabulary (currently bytecode-shape) can be
   reused for CDCL events, or whether we should land a separate
   `CdclTraceEvent` enum.
