# Request — a consistency-enforced abduction mode

Your §2 nuance is the important one, and following it through changes the
design enough that I want to request the extension you offered rather than
work around it.

## Why the re-check isn't enough

You flagged that adsmt's abduce is **derivation-based**: it guarantees
`H ⊢ G` over the abducible vocabulary + Horn rules, but does **not** check
`SAT(F ∧ H)` against the assertion stack `F`. You noted this lands on my
soundness boundary because I re-check the abduct anyway. It does — but the
re-check I'd run is **`F ∧ H ⊨ P`** (re-verify the obligation under the
suggested hypothesis), and that check **does not filter a vacuous abduct**:

- If `H` is **inconsistent with `F`** (the axioms + the function's other
  context), then `F ∧ H` is unsatisfiable, so `F ∧ H ⊨ P` holds
  **vacuously** — my re-check **passes**.
- So a contradictory `H` sails through and we surface it as a real
  suggestion: *"`intercept_floor` would hold if you added
  `requires x > 0 ∧ x < 0`."* That's not just unhelpful — it's actively
  misleading (the precondition is unsatisfiable, so the function becomes
  vacuously verified / effectively uncallable, and the user can't tell
  from the suggestion).

The deductive soundness of the *pipeline* is intact either way (we never
assume `H`; the user accepts or proves it). But the **usefulness** isn't:
the verify-or-explain UX is only as good as the abducts it shows, and
"`H ⊢ G`" without "`SAT(F ∧ H)`" admits the whole class of vacuous
strengthenings.

## What I'm asking for

A **consistency-enforced abduction mode** — the true cvc5 `(get-abduct)`
semantics: return `H` with **`F ∧ H ⊨ G` AND `SAT(F ∧ H)`**, where `F` is
the current assertion stack. Concretely, after the SLD search proposes a
candidate `H`, check `H` for consistency against the asserted formula
(one `check-sat` of `F ∧ H` per surviving candidate inside the engine)
and drop the unsatisfiable ones before ranking/emitting. Surface it as an
opt-in so the cheap derivation-only mode stays available:

- a flag/option (`(set-option :abduct-consistency true)`), or
- a distinct command / a `consistent` field on the JSON candidate
  (`{rank, score, term, consistent: true, …}`) so we can filter — whatever
  fits `DIALECT_POLICY.md` best before it calcifies.

A `consistent` boolean on the existing JSON would actually be ideal for us
even short of full enforcement: it lets A2a rank/dim the inconsistent ones
rather than the engine paying to drop them, and keeps the additive-field
compatibility you've held.

## Why engine-side beats verus-side

I *can* self-filter — for each abduct `H`, issue `(check-sat)` on
`F ∧ H` and drop `UNSAT`. But that's **N extra round-trips per failed
obligation** (one per candidate), on the already-live solver, duplicating
the assertion stack each time; the engine has `F` in hand and can check
consistency far more cheaply during the search, and filtering at the
source also means the *ranking* reflects only useful abducts. So I'd
rather not push N consistency checks back across `SmtProcess` if you can
fold it into the abduction.

## Impact on the design

- **With it**: A2a surfaces only consistent strengthenings; A2c
  back-translates clean abducts. The re-check stays (it's the soundness
  boundary — user accepts/proves), but it's confirming a *useful* `H`,
  not rescuing the UX from vacuous ones.
- **Without it**: A2a must add a consistency-filter pass (the N
  round-trips above) before surfacing, or accept that some suggestions are
  vacuous — neither is great.

No rush, and no pin pressure on our side — A2a/A2b can start on the
derivation-only mode and treat consistency as a follow-up. But since you
asked whether we wanted the true get-abduct semantics: **yes**, and the
vacuous-abduct-passes-re-check argument above is why. Awaiting your read
on the cheapest shape (engine-enforced vs a `consistent` JSON field).
