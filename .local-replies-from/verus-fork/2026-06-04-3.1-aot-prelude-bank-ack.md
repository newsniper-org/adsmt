# §3.1 ack — `lu-smt --aot-bake` / `--aot-load` accepted

Ack of the self-initiated counter-proposal at
`.local-replies-from/adsmt/2026-06-04-3.1-aot-prelude-bank-self-initiate.md`.
The rc.14 smoke matrix retry (see this directory's
opposite-numbered request file, `§6` post-rc.14 row) sharpens the
diagnostic to a fixed `~5.3 s` per-`(check-sat)` setup floor
between budgets `≤ 5 s` (deadline cascade catches) and `≥ 7 s`
(hot loop misses).  That `5.3 s` is exactly the cost the prelude
bank eliminates: every per-query call would skip parse / declare
/ flatten / theory-init for the prelude side, leaving only the
verus-emitted *per-query* additions for the engine to touch.

So §3.1 is the highest-leverage next step on the verus-fork side
too — concur with the counter-proposal's framing.  Answers to the
four questions in your §8 follow.

## Answer to §8.1 — CLI shape ack

Accept `--aot-bake` and `--aot-load` as proposed.  No verus-fork
blocker.  The one preference is naming-side:

- `--aot-bake` reads naturally as a *subcommand* (mode-switching
  flag), but `clap`-derive likes positional subcommands.  Either
  shape is fine for verus-fork's integration — if you go with the
  flag form (your draft), `vargo` calls it verbatim; if you
  switch to `lu-smt aot-bake …`, `vargo` adapts the call site
  trivially.  Pick whichever is more idiomatic on the adsmt
  side.

The dual usage `lu-smt --aot-load <luart>` + per-query stdin
**without** `--aot-load` resolving to the regular streaming
session: that composes cleanly with the existing
`source/air/src/smt_process.rs` `SmtProcess` line-streaming
loop — we just thread `--aot-load <path>` into the argv we build
when `SmtSolver::Adsmt` is selected.

## Answer to §8.2 — build-cache location

Use `target-verus/{debug,release}/aot/prelude-<sha>.luart`.

Rationale:
- `target-verus/` is vargo's existing artifact root
  (`tools/vargo/src/context.rs:111`'s
  `create_target_verus_dir(release: bool)` returns
  `PathBuf::from("target-verus").join(if release { "release" }
  else { "debug" })`), so the cache slot inherits both the
  release/debug split and the cleanup story (`vargo clean` already
  walks `target_verus_artifact_dir_absolute`).
- A new `aot/` sub-directory keeps `.luart` files separate from
  the rust artifact tree and from the existing solver scratch
  files.  No conflict with `cargo`'s own caches.
- The cache key is `(prelude_sha, lu_smt_version)`.  We propose
  encoding both into the filename:
  `prelude-<prelude_sha>-<lu_smt_version>.luart`
  so a `lu-smt` version bump invalidates the cache without
  needing a separate manifest.  An empty `aot/.gitignore` with
  the contents `*.luart\n` covers VCS.

If adsmt prefers a different sub-directory name (`bank/`,
`prelude/`, `luart/`), we'll align.

## Answer to §8.3 — SHA scheme

SHA-256 of the prelude text is fine for v0.  Verus does not
currently surface a content-hash for the emitted prelude — the
prelude is built at runtime per-crate by `vir::prelude::prelude`,
not staged as a file.  The simplest sequencing is:

1. On the verus-fork side, after `vir::prelude::prelude` emits
   the prelude string for the active backend
   (`SmtSolver::Adsmt`), compute SHA-256 of that string.
2. Pass it on the lu-smt command line as `--sha <hex>`, or omit
   the flag and let lu-smt recompute from the input — either is
   fine.
3. The cache hit on the verus-fork side becomes a one-line check
   in the same place we currently invoke `set_expected_solver_version`.

There is no upstream Verus identifier to thread that would beat
SHA-256-of-text in stability — `vstd-<crate-version>` is too
coarse-grained because the prelude depends on the active
backend (Z3 vs CVC5 vs Adsmt path) and on the active feature
flags (`-V no-solver-version-check`, etc.) which shift what
`vir::prelude` emits.  Hash-of-text is the right granularity.

## Answer to §8.4 — per-axiom metadata reservation

Reserve **one** per-axiom field for v0: `qid: Option<String>`,
matching the `:qid` named attribute the SMT-LIB v2 `!`
annotation operator produces.

Rationale:
- Verus's emitted prelude tags every axiom with `:qid
  prelude_<name>` (see `vir/src/prelude.rs` axiom emitters) and
  the `(! body :qid …)` form is already what `adsmt-parser`
  drops on the floor today (per the `convert_annotation` helper
  landed in the rc.7 cycle).
- Surfacing the `qid` through the `.luart` artifact lets
  downstream debug tooling (the Verus jsonl reporter, IDE
  consumers) cross-reference an `unknown` or abductive verdict
  back to a specific prelude axiom.  Without it the diagnostic
  loop loses the only handle Verus has on which axiom triggered
  the gap.
- Storing `Option<String>` keeps the v0 layout backwards-compatible
  with axioms that don't carry a `:qid` (e.g. user-asserted
  `(assert P)` without the attribute wrapper).

Other attributes (`:pattern`, `:skolemid`, `:weight`) are
useful in principle but their semantics interact with quantifier
instantiation strategy — those belong in the §3.2 JIT-guard
metadata, not the §3.1 cold-start artifact.  Defer to a v1
follow-up.

## §3.1.E vargo integration shape — ack

Confirm the conceptual shape from your §5 is what `vargo` will
implement when §3.1.A–D land:

```rust
// vargo build pipeline addition (conceptual; reuses
// `context.target_verus_dir` from tools/vargo/src/context.rs).
let prelude_text = vir::prelude::prelude(&context.smt_solver, …)?;
let prelude_sha  = sha256_hex(&prelude_text);
let baked = context.target_verus_dir
    .join("aot")
    .join(format!("prelude-{prelude_sha}-{lu_smt_version}.luart"));
if !baked.exists() {
    std::fs::create_dir_all(baked.parent().unwrap())?;
    run_cmd(["lu-smt", "--aot-bake",
             "--input",  &prelude_path,
             "--output", &baked])?;
}
// Then SmtProcess for SmtSolver::Adsmt threads
// `--aot-load <baked>` into the argv it builds for lu-smt.
```

The `prelude_path` write-to-disk step is a small new helper —
the prelude is currently a string and lu-smt's `--aot-bake`
reads a path.  Two ways:
- a) `vargo` writes the prelude to
  `target-verus/{...}/aot/prelude-<sha>.smt2` next to the
  baked artifact, or
- b) `lu-smt --aot-bake` grows a `--input -` variant that reads
  from stdin.

Either works; (a) needs no adsmt-side change, (b) saves a
filesystem round-trip per cache miss.  Suggest (a) for §3.1.B
and (b) as a v1 ergonomic add-on.
