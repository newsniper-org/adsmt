# Digest landed and the trace collapsed — but the wall didn't move

Re-baked on rc.34.3 (the `--jit-trace-emit-slim` bake line is unchanged)
and pinned **rc.34.3** (verus rebuilt, vstd 1690 green). The digest does
exactly what you said:

```
slim-trace:  3 565 577 B (rc.34.2)  →  99 B (rc.34.3)   [the basis is gone]
```

Verdict-independence still holds (tight-rlimit rows → `unsat` via the
consult). So size/load/compare are solved. But the consult **wall is
unchanged**, so I isolated where the ~0.42 s actually goes:

```
(1) baseline (no aot, no jit):        ~0.021 s
(2) --aot-load only (no consult):     ~0.04  s    → the bank load is cheap (~0.02 s)
(3) --aot-load + --jit-trace-load:    ~0.42  s
consult marginal cost = (3) − (2)  ≈  0.38 s
```

The trace is 99 B (load ≈ 0), so that **0.38 s is the live digest
*compute***: `jit_trace_digest()` → `canonical_clause_set` still
CNF-flattens + sorts atoms + sorts/de-dups the DIMACS of the **whole live
formula** (prelude ∪ query) on every `(check-sat)`. You dropped
`cnf_to_generators` (the polynomial encode) — real and correct — but the
canonical-clause-set construction it shared is itself O(#clauses), and a
prelude is thousands of clauses. So the digest fixed the wire and the
comparison; the canonicalization that *produces* the digest is now the
whole cost.

Net: the break-even is still ~"search > 0.38 s", unchanged from rc.34.2 —
because nothing we've done touches the per-`(check-sat)` whole-formula
canonicalization. My "clause-set hash cuts the compute" claim was half
right: it skips the polynomial layer, but not the flatten+sort+dedup
underneath it.

## The real lever — incremental / precomputed prelude canonicalization

The prelude is **fixed** across a session's `(check-sat)`s; only the
per-query delta changes (you already lean on this — the loaded trace
stays installed "the prelude is fixed; only the per-query delta
changes"). So the whole-formula canonicalization is redoing the prelude's
share every query. The lever:

- Canonicalize the prelude **once** — at `--aot-bake`, fold the prelude
  clauses into an **order-independent** combiner (e.g. XOR / sum of
  per-clause K12 hashes — the clause set is already order-independent, so
  an order-independent fold is exact) and store that partial digest in
  the bank.
- Per `(check-sat)`, the live digest = combine(prelude_partial,
  fold(query-delta clauses)) — O(#query clauses), not O(#prelude +
  #query). The exact-match stays byte-identical (same clause set → same
  fold).

That moves the consult from O(whole formula) to O(query delta), which is
where the §3.5.C cross-query prelude-reuse you flagged also wants to
land. With it, the consult is cheap even when the search isn't multi-
second.

(Honest scoping for the verus use-case: each *distinct* obligation is a
different formula → digest miss → the consult canonicalizes, misses, and
falls through, paying the cost for nothing. So even O(delta) mainly helps
the **exact re-run** case — re-verifying unchanged code against a warm
bank. That's a real case, just narrower than "every query." Not arguing
against the lever — just flagging the consult's practical reach for us is
exact re-runs, which is exactly what §3.5.J targets.)
